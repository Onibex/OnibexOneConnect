# Working with Logs

**The OneConnect solution offers two options for reviewing entity logs:**

1. **Standard OneConnect Log:** Logs are stored in the standard OneConnect logging table.
2. **TXT File in a Server Directory:** Logs are saved as a `.txt` file in a specified server file path.

The choice of where to store the logs is entirely up to the user and is configured **individually for each entity** during setup.

These logs record every process executed by an entity within transaction **`ZONT_ONECM`**, providing full traceability of operations for monitoring and debugging purposes.

---

## Configuring the Log Type by Entity

### 1. Open the OneConnect Transaction

Enter transaction **`ZONT_ONECM`**.

<img width="534" height="377" alt="image" src="https://github.com/user-attachments/assets/7c07677b-914c-4e72-bb63-a4a256e992f7" />

### 2. Select or Create an Entity

Inside the transaction, select an entity to create or modify, and follow the steps to configure the **Main Customizing Data** of the entity as explained in the [Entity Creation and Customizing manual](./01-entity-creation-and-customizing.md#configuring-main-customizing-data).

### 3. Choose the Log Type

Once you are in the **Main Customizing Data** section, locate the **Log Types** setting.

| Option | Behavior |
|---|---|
| **Standard** | No additional information is requested by the system. Logs are stored in the standard OneConnect logging table. |
| **Server File** | The system will require an existing directory path where it can store the `.txt` log files. Make sure the specified path is valid and accessible from the application server. |

## Reviewing Logs

There are two alternatives for reviewing logs, depending on the configuration set for a specific entity:

1. **Using the OneConnect Standard Log:** Logs are stored and accessed through the standard OneConnect interface.
2. **Using the Server File:** Logs are saved as a `.txt` file in a defined server directory path.

Each entity's configuration determines which method is used, and both provide full traceability of execution events.

---

## Standard Log

### 1. Open the OneConnect Transaction

Enter transaction **`ZONT_ONECM`**.

<img width="534" height="377" alt="image" src="https://github.com/user-attachments/assets/3ff438f6-ab8f-4954-94a7-772fbaa1fb83" />

### 2. Navigate to the Log Viewer

Within the OneConnect transaction, open the **"Execution"** menu at the top and select **"Display Log"**.
<img width="1089" height="435" alt="image" src="https://github.com/user-attachments/assets/55bada06-65ab-4fa4-9d72-6f4faff83a3d" />

### 3. Filter the Log Results

In the next screen, select the **Domain** of the entity you want to review, and fill in the additional parameters to filter your search as needed.

> ℹ️ Leave the **Object Log** as **`ZONI_OC`**.

In the **Message Types** field, choose the type of message you want to review. Available options include:

| Option | Description |
|---|---|
| **Errors** | Shows only error messages. |
| **Warnings** | Shows only warning messages. |
| **Success** | Shows only successful execution messages. |
| **All Messages** | Shows every log entry regardless of type. |

This allows you to narrow down the log results based on the nature of the messages generated by the entity.
<img width="1365" height="767" alt="image" src="https://github.com/user-attachments/assets/36499c22-805b-40fa-99f0-f2f6341e53c6" />

### 4. Interpret the Log Results

In the resulting list, the user can identify several key elements related to the execution of entities. The most relevant fields include:

| Field | Description |
|---|---|
| **Time and Date** | Indicates when the information was sent. |
| **User** | Specifies the user who triggered the data transmission. |
| **Entity** | The name of the executed entity. |
| **Object** | The real-time object used to send the information (if applicable). |
| **Table** | The name of the table involved in the transmission. |
| **Key** | The key value from the transmitted table. |
| **Message Text** | Displays the result or outcome message of the execution. |
| **RFC Destination** | Indicates the endpoint to which the information was sent. |
